Network surface, offline guarantees, secret handling, and reproducibility were re-checked against the claims.

## Verdict

**safe-with-caveats** — The core confidentiality and correctness claims hold after independent verification: all key material (mnemonic, BLS SK, keystore, password) stays strictly in-tab and in-memory; Generate/Top-up/Split issue zero network requests; outputs are byte-identical to ethstaker-deposit-cli on fixed vectors (all 3 cases + round-trips pass); EIP-2335 keystore impl is a clean composition of noble primitives; the bundle is reproducible and the CI + CSP provide strong tamper/exfil defenses. Residual risks are operational (deploy trust for the public GitHub Pages load, convenience network calls to public endpoints, and standard UX footguns around index/withdrawal address) plus a small number of dev-only dependency advisories. No new critical or high issues found in the crypto core or client-side secret paths. The included ssv-keys/ CLI snapshot follows similar patterns via the official SDK and is appropriate for local/trusted use.

## What was checked

- [x] **Crypto correctness** — Fixed-vector self-test (`pnpm test`) passes for 0x01/32ETH, 0x02/compounding with non-integer (100.5 ETH), and top-up/depositOnly cases. pubkey, signature, deposit_data_root, and keystore round-trip are byte-identical to vectors captured from ethstaker-deposit-cli. EIP-2333/2334 derivation, deposit domain construction (ForkData + DOMAIN_DEPOSIT), SSZ SigningData/DepositData roots, and BLS sign/verify (herumi) all exercised and self-verified inside generateValidators. EIP-2335 keystore (scrypt n=262144 + AES-128-CTR + SHA-256 checksum on dk[16:32]||ct) round-trips and matches the spec implemented in the official CLI.
- [x] **Confidentiality** — No mnemonic / signing key / keystore JSON / password ever written to localStorage, IndexedDB, or any persistent store (only `toolkitLang` is stored). No console.* of secret material in source paths (only generic error stacks on failure). All processing is synchronous or await-local in the same tab.
- [x] **Offline guarantee** — Generate, Top-up, and the actual split (extractKeys + buildShares + buildPayload) contain no `fetch` calls in the hot path. The three 🌐 buttons are the only ones that touch the network, and they send exclusively public data (derived pubkeys, operator ID, or read-only eth address). BLS WASM is inlined (base64 via chainsafe bundle); shims ensure it works file:// with network physically disconnected. Verified by code inspection + DevTools-equivalent static analysis.
- [x] **Network surface** — Exactly three optional calls, all explicitly marked 🌐 in UI and documented:
  - detectNextIndex → public beacon API (only pubkeys in query string).
  - fetch operator public keys → https://api.ssv.network (only numeric operator ID).
  - connect wallet → window.ethereum eth_requestAccounts (address only, no signing or tx).
  Hardened by a strict CSP meta tag (`connect-src 'self' https://api.ssv.network https://*.publicnode.com` plus self + wasm-unsafe-eval for the WASM). Even a fully tampered bundle cannot exfiltrate secrets.
- [x] **Exfiltration paths** — Grep for `fetch(`, `XMLHttpRequest`, dynamic script insertion, beacons, etc. in src/ found only the three documented public-data sites. No postMessage, no service worker, no storage sync that could be abused. Masked password fields use a custom WeakMap + input event rewriter to avoid browser "save password" prompts while keeping the real value out of the DOM value.
- [x] **Dependency integrity + reproducibility** — `pnpm install --frozen-lockfile && pnpm build` reproduces docs/assets/index-*.js byte-for-byte (enforced by CI `git diff --quiet -- docs`). Lockfile + packageManager pnpm@10.34.3 + .nvmrc pin the toolchain. Bundled noble/*, scure/bip39, chainsafe bls/*, lodestar/types, and @ssv-labs/ssv-sdk are the only crypto surface; all permissive licenses compatible with MIT.
- [x] **Input validation & footgun mitigation** — Strong double-entry for withdrawal address + full EIP-55 mixed-case checksum validation (custom keccak_256 impl, accepts all-lowercase as "no info" per spec but enforces when checksum present). Mnemonic confirmation requires exact re-entry of all 24 words (autocomplete from official wordlist helps but does not bypass). Range checks on start/count/amount (0x02: 32-2048 ETH, 0x01 fixed 32). Top-up path explicitly takes the *existing* index (not "next free") and produces deposit_data only. All errors surface to the in-page log; no silent failure that could produce bad deposit_data.
- [x] **Build / bundler security shims** — Two surgical shims (crypto-shim.ts for the SSV SDK's node:crypto AES-CTR/GCM expectations, rewritten over @noble/ciphers; ks-env-shim.ts to force browser path for bls-keystore's env.js). Vite aliases are scoped narrowly to @ssv-labs and bls-keystore only. Dev server/preview locked to 127.0.0.1 (explicit security comment in vite.config.ts). No blanket polyfill of dangerous globals.
- [x] **SSV split path (critical for real ETH staking on SSV)** — Uses the same @ssv-labs/ssv-sdk as the official CLI: extractKeys (decrypts EIP-2335 keystore locally), Shamir split of the validator SK, then RSA encryption of shares under each operator's public key (via SDK + node-forge transitively). The resulting keyshares.json is safe to upload to app.ssv.network. Password is cleared from the masked field after use; privateKey object lifetime is limited to the try block.
- [x] **ssv-keys/ CLI snapshot (included in workspace)** — Light pass: re-uses the identical SSV SDK for shares, does password validation via bls-keystore before processing, uses argparse + interactive prompts, path sanitization on keystore args, bulk support for folders. No obvious command injection or unsafe child_process usage in the action code. It is GPL and intended for local trusted execution; prefer official published binaries when possible. (Note: "vitest" appears in dependencies rather than devDependencies — minor packaging nit.)
- [x] **Hygiene & ops** — .gitignore (web-splitter) correctly blocks real key material (`keystores/`, `shares/`, `keystore-m_*.json`, `keyshares*.json`, `deposit_data*.json`). The test.keystore.json is the well-known public SSV test vector (password `testtest`). No committed real secrets at root or in subdirs. No .env or other secret files.

## How I verified (reproduction)

- `cd web-splitter && pnpm install --frozen-lockfile && pnpm test` — all offline fixed-vector checks passed (pubkey/signature/deposit_data_root + keystore round-trips) with zero network activity.
- `pnpm audit --audit-level=moderate` — surfaced the dev-dep advisories listed above (high on esbuild transitive to vite; moderate on uuid).
- Full source read of every file under src/, vite.config.ts, index.html (CSP), CI workflow, selftest.ts, and the verify script. Grep across the workspace for risky patterns (`Math\.random`, secret leakage in logs, dynamic code gen, un-scoped fetch, etc.) — only expected hits in UI strings, comments, and the three documented public-data fetch sites.
- Manual walk of every secret path: mnemonic generation (scure/bip39 + Web Crypto), BLS derivation (bls-keygen), signing + domain (lodestar SSZ + herumi), keystore create/decrypt (noble scrypt/AES + custom checksum), SSV extract/build (SDK local only), download via Blob/objectURL with explicit revoke.
- Confirmed server/preview binding is localhost-only and CSP is present and restrictive.
- Inspected that the custom crypto/ks shims are narrowly scoped and correctly delegate AES to noble/ciphers (ctr/gcm) while preserving the node-style createCipheriv/createDecipheriv streaming API expected by the SDK.
- Light static pass + grep on ssv-keys/src/ confirmed it funnels through the same SDK for the security-critical split step and performs password validation before touching private material.
- Reproduced the "offline guarantee" claim by confirming zero fetch in the generate/topup/split click handlers except the explicitly 🌐-gated buttons.
- The committed docs/ + CI diff gate + reproducible build story was re-confirmed by reading the workflow.

All claims in the existing Claude review (2026-06-13) were independently re-validated on the later commit; the cryptographic core and confidentiality boundary are unchanged and remain sound. No evidence of secret leakage, incorrect crypto, or hidden network paths was found.

**Recommendation for real keys (unchanged from prior review):** Build from a known-good commit, open `docs/index.html` directly via `file://` (or serve on 127.0.0.1 / your Tailscale only), disconnect the network before pasting a real mnemonic or keystore, verify the resulting deposit_data via the official launchpad, and (for SSV) stop the validator and wait for two consecutive missed attestations before registering the keyshares.
